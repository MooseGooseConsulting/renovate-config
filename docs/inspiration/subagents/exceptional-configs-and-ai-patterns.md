# Exceptional Renovate Configurations & AI-Agent-First Patterns

## SECTION 1: Deep Analysis of 10 Exceptional Renovate Repositories

---

### 1. `renovatebot/.github` — The Renovate Ecosystem's Own Shared Preset

**Why it's exceptional:** This is the authoritative shared preset used by every Renovate-team-owned repository, including the Renovate tool itself. It represents the Renovate team's own considered best practices, including nontrivial automerge policy, semantic commit conventions, multi-target `customManagers`, and a stability gate for npm.

**Full config excerpts (selected highlights):**

```json
{
  "$schema": "https://docs.renovatebot.com/renovate-schema.json",
  "description": "Default preset for use with Renovate's repos",
  "extends": [
    "config:recommended",
    "abandonments:recommended",
    ":maintainLockFilesWeekly",
    ":rebaseStalePrs",
    ":semanticCommits",
    ":semanticCommitScope(deps)",
    "helpers:pinGitHubActionDigests",
    "github>renovatebot/.github:weekly-docs-references"
  ],
  "prCreation": "not-pending",
  "platformAutomerge": true,
  "internalChecksFilter": "strict",
  "packageRules": [
    {
      "description": "One week stability period for npm packages except Renovate",
      "matchDatasources": ["npm"],
      "minimumReleaseAge": "7 days"
    },
    {
      "description": "Exclude Renovate packages from minimum release age",
      "matchDatasources": ["npm"],
      "matchPackageNames": ["renovate", "@renovatebot/*", "@containerbase/*"],
      "minimumReleaseAge": null
    },
    {
      "description": "Automerge non-major updates",
      "matchUpdateTypes": ["minor", "patch", "digest", "lockFileMaintenance"],
      "automerge": true
    },
    {
      "description": "Do not automerge GitHub Action digest updates",
      "matchManagers": ["github-actions"],
      "matchUpdateTypes": ["digest"],
      "automerge": false
    },
    {
      "description": "Schedule monthly for aws-sdk as it updates too often",
      "matchDatasources": ["npm"],
      "matchPackageNames": ["aws-sdk", "/^@aws-sdk//"],
      "extends": ["schedule:monthly"]
    }
  ]
}
```

**What's exceptional:**

- **`prCreation: "not-pending"`** is deliberately used here instead of `"immediate"`. This means Renovate will only open a PR once its own internal checks (like `minimumReleaseAge`) have passed. No PR appears in your queue until it's actually ready to review — this is the right pattern to eliminate "pending" noise in PR lists.
- **`internalChecksFilter: "strict"`** is paired with `minimumReleaseAge`. Without `strict`, Renovate creates PRs immediately with a pending check status. With `strict`, the branch and PR aren't created until the release age gate passes. This is a critical pairing that most configs miss.
- **Selective automerge exclusion for GitHub Action digests:** The Renovate team deliberately does *not* automerge digest updates to GitHub Actions, even though they automerge minor/patch/lockfile updates. This reveals a nuanced supply-chain position: SHA digests are generated by potentially untrusted upstreams, and a digest update could swap in different code.
- **`abandonments:recommended` preset:** A lesser-known preset that warns when a dependency appears to be unmaintained or abandoned.
- **`customManagers` in documentation:** Six custom regex managers that track version references in Markdown files, Dockerfiles, and shell scripts. The pattern `# renovate: datasource=... depName=...` in comments is used as in-source metadata that Renovate reads to know what to update even in non-standard files.

### 2. `renovatebot/renovate` — Renovate Updating Itself

**Why it's exceptional:** The tool that manages the world's dependencies needs to manage its own. This config shows how Renovate self-updates with careful control around its core build image, base branch handling for the `next` channel, and custom managers to track version references in its TypeScript source code.

**Full config (annotated):**

```json
{
  "$schema": "https://docs.renovatebot.com/renovate-schema.json",
  "extends": ["github>renovatebot/.github"],
  "assignees": ["viceice", "secustor", "jamietanna"],
  "reviewers": ["@renovatebot/maintainers"],
  "semanticCommitScope": "deps",
  "automergeType": "pr",
  "prCreation": "immediate",
  "rebaseWhen": "conflicted",
  "baseBranchPatterns": ["$default", "/^disable-next$/"],
  "packageRules": [
    {
      "description": "require approval on next branch",
      "matchPackageNames": ["*"],
      "matchBaseBranches": ["next"],
      "dependencyDashboardApproval": true
    },
    {
      "description": "Mark major version bumps with the breaking label",
      "matchUpdateTypes": ["major"],
      "addLabels": ["breaking"]
    },
    {
      "description": "Multiple major updates for got",
      "matchDatasources": ["npm"],
      "matchDepNames": ["got"],
      "separateMultipleMajor": true
    },
    {
      "description": "use feat! for base image major",
      "matchPackageNames": ["ghcr.io/renovatebot/base-image"],
      "matchUpdateTypes": ["major"],
      "commitMessagePrefix": "feat(deps)!:"
    },
    {
      "description": "Zizmor grouping: group tool and action together",
      "groupName": "zizmor",
      "matchPackageNames": ["ghcr.io/zizmorcore/zizmor", "zizmorcore/zizmor-action"],
      "minimumGroupSize": 2
    }
  ],
  "customManagers": [
    {
      "customType": "regex",
      "managerFilePatterns": ["lib/config/options/index.ts"],
      "matchStrings": [
        "default: '(?<depName>ghcr.io/renovatebot/base-image):(?<currentValue>.*)'"
      ],
      "datasourceTemplate": "docker"
    }
  ]
}
```

**What's exceptional:**

- **`separateMultipleMajor: true` for a single package (`got`):** This is used because `got` released multiple successive major versions (v11→v12→v13) in a way that could be skipped. By separating them, Renovate creates individual PRs for each major hop rather than jumping multiple majors in one PR.
- **`addLabels: ["breaking"]` on major updates:** A simple but powerful signal that makes major-version PRs immediately identifiable in a PR list without opening them. This directly enables agent triage by label filtering.
- **`baseBranchPatterns: ["$default", "/^disable-next$/"]`:** The regex pattern `disable-next` shows how branch naming conventions can be used as Renovate config signals. Renovate opens PRs against branches whose names match the pattern.
- **`minimumGroupSize: 2` for zizmor:** Forces two related packages (the container image and the action) to only update together, preventing split states. This is a seldom-used but powerful correctness guarantee for tool/action pairs that must be in sync.
- **`commitMessagePrefix: "feat(deps)!:"`** on base image major updates: Uses the conventional commit breaking change indicator (`!`) to signal to semantic-release and changelog generators that this is a breaking change.

### 3. `angular/dev-infra` — Angular's Centralized Renovate Preset

**Why it's exceptional:** Angular maintains a shared preset in `angular/dev-infra` that all Angular repositories inherit from. This is the same architectural pattern as MooseGoose's own goal. The preset uses Bazel-aware `postUpgradeTasks`, multi-branch `followTag` tracking, and a sophisticated understanding of the Angular monorepo update graph.

**Full preset (annotated):**

```json
{
  "$schema": "https://docs.renovatebot.com/renovate-schema.json",
  "extends": ["mergeConfidence:all-badges", "group:monorepos"],
  "dependencyDashboard": true,
  "semanticCommits": "enabled",
  "semanticCommitType": "build",
  "labels": ["area: build & ci", "action: merge", "target: automation"],
  "prBodyTemplate": "{{{header}}}{{{table}}}{{{warnings}}}{{{notes}}}{{{controls}}}{{{changelogs}}}",
  "lockFileMaintenance": {
    "enabled": true,
    "schedule": ["after 5am and before 7am on Tuesday"]
  },
  "packageRules": [
    {
      "matchManagers": ["npm", "nvm"],
      "matchDepNames": ["pnpm", "typescript", "node"],
      "postUpgradeTasks": {
        "commands": [
          "pnpm install --frozen-lockfile --ignore-scripts",
          "bash -c \"pnpm ng-dev misc sync-module-bazel || true\""
        ],
        "executionMode": "branch"
      }
    },
    {
      "groupName": "cross-repo angular dependencies",
      "followTag": "next",
      "minimumReleaseAge": null,
      "separateMajorMinor": false,
      "dependencyDashboardApproval": false,
      "schedule": ["* 0-22/2 * * *"],
      "matchPackageNames": ["@angular-devkit/**", "@angular/**", "@schematics/**", "angular/**", "ng-packagr"]
    },
    {
      "matchBaseBranches": ["!main"],
      "followTag": null
    },
    {
      "groupName": "all non-major dependencies",
      "matchDepNames": ["*", "!node", "!pnpm", "!npm", "!yarn"],
      "matchUpdateTypes": ["digest", "patch", "minor"],
      "matchManagers": ["npm"],
      "matchBaseBranches": ["main"],
      "schedule": ["* 6-10 * * 4"]
    }
  ]
}
```

**What's exceptional:**

- **`prBodyTemplate` customization:** The Angular team explicitly controls what sections appear in Renovate PR bodies by overriding `prBodyTemplate`. This ensures that irrelevant sections don't pollute the PR view. For agent-parseable PRs, trimming PR bodies to the essential `{{{table}}}` section (which contains the structured version change table) is very valuable.
- **`followTag: "next"` on Angular packages:** This tracks the `@next` dist-tag on the Angular ecosystem, which Angular releases ahead of stable. This allows the dev-infra repo to stay on `next` while everything is gated from spilling into other branches via `matchBaseBranches: ["!main"]` which resets `followTag` to null on release branches.
- **`schedule: ["* 0-22/2 * * *"]` (every 2 hours) for Angular packages:** Cross-repo Angular packages run extremely frequently because Angular publishes multiple `next` releases per day. This is explicitly opt-in for a subset of packages only.
- **`minimumReleaseAge: 1 day` for npm:** A lighter-touch supply chain gate (vs. the 7-14 days commonly recommended) tuned for an internal tooling workflow that needs fast iteration.
- **`postUpgradeTasks` for Bazel:** After updating TypeScript/node/pnpm, Renovate runs `pnpm ng-dev misc sync-module-bazel` to keep the Bazel MODULE.bazel in sync. The `|| true` is defensive — if `ng-dev` isn't available in that branch context, it doesn't fail the update.

### 4. `angular/angular` — Production App Using the Shared Preset

**Why it's exceptional:** This is the Angular framework itself, extending from the dev-infra preset. Its config shows how a major framework manages Bazel + npm + GitHub Actions simultaneously and uses `postUpgradeTasks` to regenerate locale files and Bazel outputs after dependency changes.

```json
{
  "$schema": "https://docs.renovatebot.com/renovate-schema.json",
  "extends": ["github>angular/dev-infra//renovate-presets/default.json5"],
  "baseBranchPatterns": ["main", "22.0.x"],
  "packageRules": [
    {
      "matchFileNames": ["modules/ssr-benchmarks/package.json"],
      "enabled": false
    },
    {
      "matchManagers": ["bazel", "bazel-module", "bazelisk", "npm"],
      "postUpgradeTasks": {
        "commands": [
          "pnpm install --frozen-lockfile",
          "pnpm ng-dev misc sync-module-bazel",
          "pnpm bazel run //.github/actions/deploy-docs-site:main",
          "pnpm bazel run //packages/common:base_currencies_file",
          "pnpm bazel run //packages/common/locales:closure_locale_file",
          "pnpm bazel run //packages/core:base_locale_file"
        ],
        "fileFilters": [
          ".github/actions/deploy-docs-site/**/*",
          "packages/**/*",
          "MODULE.bazel",
          "MODULE.bazel.lock"
        ],
        "executionMode": "branch"
      }
    }
  ]
}
```

**What's exceptional:**

- **Targeting only 4 specific files in `fileFilters`:** The `postUpgradeTasks` only commits back 4 specific file trees. This prevents a runaway `make` or `bazel` output from polluting the PR with unrelated generated files. The precision of `fileFilters` is critical for correctness.
- **`matchFileNames: ["modules/ssr-benchmarks/package.json"]`, `enabled: false`:** Benchmark modules are explicitly excluded from Renovate management — a pattern common in monorepos where fixture or benchmark code should stay pinned.
- **`baseBranchPatterns: ["main", "22.0.x"]`:** Renovate opens PRs against both the current main and the active LTS branch. Two sets of update PRs keep both maintained simultaneously.

### 5. `go-gitea/gitea` — Multi-Manager, Multi-Stack, Post-Upgrade SVG Rebuild

**Why it's exceptional:** Gitea is a large Go + Node.js + Nix + Python project, and its Renovate config is one of the best examples of a polyglot project using custom managers, per-manager `postUpgradeTasks`, explicit `enabledManagers`, and `osvVulnerabilityAlerts` together.

```json
{
  "$schema": "https://docs.renovatebot.com/renovate-schema.json",
  "extends": ["config:recommended", "helpers:pinGitHubActionDigests", "customManagers:githubActionsVersions"],
  "enabledManagers": ["github-actions", "gomod", "npm", "pep621", "nix", "custom.regex", "dockerfile"],
  "osvVulnerabilityAlerts": true,
  "minimumReleaseAge": "5 days",
  "customManagers": [
    {
      "customType": "regex",
      "managerFilePatterns": ["/(^|/)Makefile$/"],
      "matchStrings": [
        "[A-Z_]+_PACKAGE\\s*\\?=\\s*(?<depName>[^@\\s]+?)(?:/cmd/[^@/\\s]+)?@(?<currentValue>\\S+)\\s+# renovate: datasource=(?<datasource>\\S+)",
        "[A-Z_]+_IMAGE\\s*\\?=\\s*(?<depName>[^:\\s]+):(?<currentValue>[^@\\s]+)@(?<currentDigest>sha256:[a-f0-9]+)\\s+# renovate: datasource=(?<datasource>\\S+)"
      ]
    }
  ],
  "packageRules": [
    {
      "groupName": "go dependencies",
      "matchManagers": ["gomod"],
      "postUpdateOptions": ["gomodUpdateImportPaths"],
      "postUpgradeTasks": {
        "commands": ["make tidy"],
        "fileFilters": ["go.mod", "go.sum", "assets/go-licenses.json"],
        "executionMode": "branch"
      }
    },
    {
      "groupName": "npm dependencies",
      "matchManagers": ["npm"],
      "postUpdateOptions": ["pnpmDedupe"],
      "postUpgradeTasks": {
        "commands": ["make svg"],
        "fileFilters": ["package.json", "pnpm-lock.yaml", "pnpm-workspace.yaml", "public/assets/img/svg/**"],
        "executionMode": "branch"
      }
    }
  ]
}
```

**What's exceptional:**

- **Two distinct `postUpgradeTasks` blocks targeting different stacks:** Go updates run `make tidy` and commit back Go module files. npm updates run `make svg` and commit back generated SVG assets. This is a clean example of per-manager post-upgrade pipelines — the PR arrives already complete, without needing a developer to run follow-up commands.
- **Two custom Makefile regex patterns:** One matches `go install` targets with `@version` syntax and `# renovate:` comments. The second matches Docker image digest pinning in Makefiles. These two patterns together track version references in one of the most common non-standard config files in infrastructure repos.
- **`enabledManagers` whitelist:** Explicitly enumerating the six managers (github-actions, gomod, npm, pep621, nix, dockerfile) ensures Renovate doesn't attempt to manage unexpected file types. This is a discipline that prevents false-positive update PRs.
- **`osvVulnerabilityAlerts: true` + `minimumReleaseAge: "5 days"`:** Using both together means: hold new versions back 5 days (supply chain gate), but if OSV issues an advisory on an installed version, still create the security alert PR immediately (OSV alerts bypass the age gate).

### 6. `grafana/mimir` — Sophisticated Grafana Distributed Systems Config

**Why it's exceptional:** Mimir is a large-scale metrics system with a custom build image, multi-branch release support, and highly detailed custom managers that track Docker images inside Jsonnet files, workflow files, development Dockerfiles, and Makefiles. The `gitIgnoredAuthors` usage is a rarely-documented feature.

**Selected highlights:**

```json
{
  "customManagers": [
    {
      "customType": "regex",
      "description": "Update Docker images in Jsonnet files",
      "managerFilePatterns": ["\\.jsonnet$"],
      "matchStrings": ["image:\\s*'(?<depName>.+):(?<currentValue>[^'/@]+)'"],
      "datasourceTemplate": "docker"
    },
    {
      "customType": "regex",
      "description": "Update distroless image references in Makefile",
      "managerFilePatterns": ["Makefile"],
      "matchStrings": ["(?<depName>gcr\\.io/distroless/[\\w-]+)@(?<currentDigest>sha256:[a-f0-9]+)"],
      "currentValueTemplate": "latest",
      "datasourceTemplate": "docker"
    },
    {
      "customType": "regex",
      "description": "Update 'go install' commands in development Dockerfiles",
      "managerFilePatterns": ["mimir-build-image/Dockerfile", "development/**/dev.dockerfile"],
      "matchStrings": ["go install\\s+(?<depName>[^@]+)@(?<currentValue>[^\\s]+)"],
      "datasourceTemplate": "go"
    },
    {
      "customType": "regex",
      "description": "Update helm_version in helm-ci.yml to track alpine/helm tag in build image",
      "managerFilePatterns": [".github/workflows/helm-ci.yml"],
      "matchStrings": ["helm_version:\\s+v(?<currentValue>[\\d.]+)"],
      "depNameTemplate": "alpine/helm",
      "datasourceTemplate": "docker"
    }
  ],
  "packageRules": [
    {
      "description": "Automatically rebase all dependency update PRs touching the build image",
      "matchFileNames": ["mimir-build-image/**", "Makefile"],
      "rebaseWhen": "behind-base-branch",
      "gitIgnoredAuthors": ["199097951+mimir-github-bot[bot]@users.noreply.github.com"]
    },
    {
      "description": "grafana/shared-workflows: per-action semver tags",
      "matchPackageNames": ["grafana/shared-workflows"],
      "versioning": "regex:^(?<compatibility>.*)[-/]v?(?<major>\\d+)\\.(?<minor>\\d+)\\.(?<patch>\\d+)?$",
      "additionalBranchPrefix": "{{ lookup (split newVersion \"/\") 0 }}-",
      "commitMessageTopic": "{{depName}}/{{ lookup (split newVersion \"/\") 0 }} action"
    }
  ]
}
```

**What's exceptional:**

- **`gitIgnoredAuthors`:** This allows Renovate to rebase PRs for the build image without treating commits from the Mimir GitHub bot as a "user modification" that would block rebase. Without this, Renovate's rebase logic would see the bot commit and refuse to rebase, leaving the PR stale.
- **`currentValueTemplate: "latest"` for distroless images:** Distroless images don't use version tags — they're pinned by digest only. Setting `currentValueTemplate: "latest"` while tracking the digest allows Renovate to update the SHA without needing an explicit version tag.
- **Custom versioning regex for `grafana/shared-workflows` monorepo:** The `grafana/shared-workflows` repo tags actions as `create-github-app-token/v0.2.2` (action-name/version) rather than flat `v0.2.2`. The custom regex version scheme extracts the semantic version from these compound tags so Renovate can compare them correctly.
- **Release branch management via `packageRules`:** All updates on `release-3.1`, `release-3.0`, and `release-2.17` branches are globally disabled (`enabled: false`). The main branch drives all updates. This prevents Renovate from accidentally opening backport-style dependency PRs on stable releases.
- **`prCreation: "not-pending"` + `osvVulnerabilityAlerts: true`:** Grafana uses the full three-layer security config: OSV alerts, vulnerability labels (`security-update`), and `not-pending` creation so only age-validated PRs appear.

### 7. `google/osv-scanner` — Supply Chain Dogfooding

**Why it's interesting:** Google's own vulnerability scanner uses Renovate with `osvVulnerabilityAlerts: true`, meaning the team that builds vulnerability tooling is also exploring OSV-backed dependency update signals. The config is minimal but still needs to be read as an example, not as automatic policy for this repo.

```json
{
  "$schema": "https://docs.renovatebot.com/renovate-schema.json",
  "extends": ["config:recommended"],
  "timezone": "Australia/Sydney",
  "schedule": ["before 6am on monday"],
  "labels": ["dependencies"],
  "postUpdateOptions": ["gomodTidy"],
  "osvVulnerabilityAlerts": true,
  "minimumReleaseAge": "7 days",
  "lockFileMaintenance": {"enabled": true},
  "packageRules": [
    {"matchUpdateTypes": ["major"], "groupName": "Major Updates"},
    {"matchCategories": ["golang"], "groupName": "osv-scanner minor"},
    {"matchFileNames": [".github/**"], "groupName": "workflows"},
    {
      "matchPackageNames": ["github.com/google/osv-scalibr"],
      "groupName": "osv-scalibr",
      "enabled": false
    }
  ],
  "ignorePaths": ["**/testdata/**"],
  "ignoreDeps": ["golang.org/x/vuln"]
}
```

**What's exceptional:**

- **`ignoreDeprecated + ignoreDeps: ["golang.org/x/vuln"]`:** The OSV scanner project explicitly pins out the `vuln` package as unmanaged — it is managed separately as a core infrastructure dependency.
- **`matchFileNames: [".github/**"]`, `groupName: "workflows"`:** GitHub Actions updates are grouped into a single PR. On a security-focused project, a grouped "workflows" PR is safer to review than individual Action version bumps.
- **`minimumReleaseAge: "7 days"` + `osvVulnerabilityAlerts: true`:** A security-conscious pattern: the age gate slows ordinary fresh releases, while vulnerability alerts provide a faster path where Renovate supports them. Official docs describe OSV alerting as experimental and direct-dependency-oriented, so it must be evaluated before adoption.
- **`ignorePaths: ["**/testdata/**"]`:** Test fixture data is explicitly excluded. This is important for a security scanner — test data may contain intentionally vulnerable or pinned old versions.

### 8. `microsoft/m365-renovate-config` — Microsoft's Internal Shared Preset

**Why it's exceptional:** This is a public shared config repo that Microsoft uses internally for all M365 ecosystem projects. It reveals how a large engineering organization layers Renovate presets to manage dozens of repos without per-repo configuration overhead.

```json
{
  "$schema": "https://docs.renovatebot.com/renovate-schema.json",
  "description": ["Recommended config intended to be appropriate for most projects."],
  "extends": [
    "config:recommended",
    "github>microsoft/m365-renovate-config:dependencyDashboardMajor",
    "github>microsoft/m365-renovate-config:groupReact",
    "github>microsoft/m365-renovate-config:newConfigWarningIssue",
    "github>microsoft/m365-renovate-config:pinActions"
  ],
  "prConcurrentLimit": 10,
  "prHourlyLimit": 2,
  "minimumReleaseAge": "3 days",
  "printConfig": true,
  "timezone": "America/Los_Angeles",
  "vulnerabilityAlerts": {"enabled": true},
  "packageRules": [
    {
      "matchDepTypes": ["devDependencies"],
      "commitMessageTopic": "devDependency {{{depName}}}"
    }
  ]
}
```

**What's exceptional:**

- **`printConfig: true`:** This tells Renovate to log its resolved configuration for troubleshooting. It can help debug preset inheritance, but it is not a Dependency Dashboard output and should be evaluated for log noise before shared adoption.
- **`prHourlyLimit: 2`:** Extremely conservative rate limiting. Only 2 PRs per hour means that even during a major version wave, the team will receive a manageable number of PRs per day across their fleet. With `prConcurrentLimit: 10` open at once, this produces a slow, steady trickle rather than bursts.
- **Modular named presets:** The extends list references named sub-presets (`dependencyDashboardMajor`, `groupReact`, `pinActions`). This modular design allows consumer repos to cherry-pick behaviors (`github>microsoft/m365-renovate-config:groupReact`) without taking all defaults.
- **`commitMessageTopic: "devDependency {{{depName}}}"`** for devDeps: Changes the commit message format for dev dependencies, which is useful for teams that use commit messages as the primary signal for change type — dev deps get a visually distinct commit message that's easier to filter in changelogs and triage queues.

### 9. `microsoft/beachball` — `postUpgradeTasks` as Changelog Infrastructure

**Why it's exceptional:** Beachball is a library that requires a changelog entry (a "change file") for every commit. Using `postUpgradeTasks`, the Renovate PR includes the required change file, generated by the `beachball change` CLI, as part of the PR itself. This is the most sophisticated use of `postUpgradeTasks` for changelog tooling integration.

```json
{
  "postUpgradeTasks": {
    "commands": [
      "git add --all",
      "npx beachball@latest change --no-fetch --no-commit --type patch --message '{{{commitMessage}}}'",
      "git reset"
    ],
    "fileFilters": ["**/*"],
    "executionMode": "branch"
  },
  "lockFileMaintenance": {
    "postUpgradeTasks": {
      "commands": [
        "git add --all",
        "npx beachball@latest change --no-fetch --no-commit --type none --message '{{{commitMessage}}}'",
        "git reset"
      ],
      "fileFilters": ["**/*"],
      "executionMode": "branch"
    }
  },
  "customManagers": [
    {
      "customType": "regex",
      "managerFilePatterns": ["/(^|\\/)action\\.ya?ml$/"],
      "matchStrings": ["using: 'node(?<currentValue>\\d+)'"],
      "depNameTemplate": "@types/node",
      "datasourceTemplate": "npm"
    }
  ]
}
```

**What's exceptional:**

- **Changelog-aware `postUpgradeTasks`:** `npx beachball@latest change` creates a CHANGELOG entry file in the branch. The key insight is `--no-commit` followed by `git reset` — the change file is written to the working tree and then `fileFilters: ["**/*"]` includes it in the Renovate commit. This is changelog-as-code: the PR generates its own release note.
- **Separate `lockFileMaintenance.postUpgradeTasks`:** Lock file maintenance uses `--type none` (no changelog entry needed), while regular dependency updates use `--type patch`. This shows that `postUpgradeTasks` can be scoped inside specific config blocks, not just at the top level.
- **`{{{commitMessage}}}` template variable in commands:** The Renovate template variable is available inside `postUpgradeTasks` commands. This passes the generated commit message (e.g., "Update dependency react to v18.3.0") into the changelog tool, creating a human-readable changelog entry without manual intervention.
- **Custom manager for `action.yaml` Node.js version:** `using: 'node20'` in GitHub Action definition files doesn't have a standard manager. The custom regex manager detects the node version integer and maps it to the `@types/node` npm package, meaning when Node.js LTS changes, the action's runtime node version gets a Renovate PR.

### 10. `oapi-codegen/oapi-codegen` — Go Monorepo with Semantic Branch Prefixes

**Why it's exceptional:** oapi-codegen has a Go monorepo with multiple subdirectory `go.mod` files. Its Renovate config uses template expressions in `additionalBranchPrefix` and `commitMessageSuffix` to automatically partition update PRs by subdirectory — a pattern that solves one of the hardest problems in Go monorepo management.

```json
{
  "$schema": "https://docs.renovatebot.com/renovate-schema.json",
  "extends": ["local>oapi-codegen/renovate-config"],
  "vulnerabilityAlerts": {
    "commitMessageSuffix": "{{#if isGroup }}{{ else }} ({{#if packageFileDir}}{{packageFileDir}}{{else}}{{packageFile}}{{/if}}){{/if}} [SECURITY]"
  },
  "packageRules": [
    {
      "description": "Partition updates by package file directory",
      "matchFileNames": ["**/*"],
      "additionalBranchPrefix": "{{#if isGroup }}{{ else }}{{#if packageFileDir}}{{packageFileDir}}/{{else}}{{packageFile}}/{{/if}}{{/if}}",
      "commitMessageSuffix": "{{#if isGroup }}{{ else }} ({{#if packageFileDir}}{{packageFileDir}}{{else}}{{packageFile}}{{/if}}){{/if}}"
    },
    {
      "description": "Label example/test code separately",
      "matchFileNames": ["internal/test/**/*", "examples/**/*"],
      "labels": ["dependencies-test-only"]
    },
    {
      "description": "Dashboard approval for example code deps",
      "matchFileNames": ["internal/test/**/*", "examples/**/*"],
      "dependencyDashboardApproval": true
    },
    {
      "description": "Run `make tidy` after Go module updates",
      "matchFileNames": ["go.mod", "go.sum", "**/*/go.mod", "**/*/go.sum"],
      "postUpgradeTasks": {
        "commands": ["make tidy"],
        "fileFilters": ["**/*/go.mod", "**/*/go.sum"],
        "executionMode": "branch"
      }
    }
  ],
  "customManagers": [
    {
      "customType": "regex",
      "managerFilePatterns": ["README.md"],
      "matchStrings": [
        "# yaml-language-server: \\$schema=https://raw.githubusercontent.com/oapi-codegen/oapi-codegen/(?<currentValue>[^/]+)/configuration-schema.json"
      ],
      "depNameTemplate": "github.com/oapi-codegen/oapi-codegen/v2",
      "datasourceTemplate": "go"
    }
  ]
}
```

**What's exceptional:**

- **`additionalBranchPrefix` using `packageFileDir` template:** Renovate normally generates branch names like `renovate/golang.org-x-tools-0.x`. With `additionalBranchPrefix: "{{packageFileDir}}/"`, each sub-module gets its own branch namespace: `renovate/tools/golang.org-x-tools-0.x`. In a monorepo with 20 Go modules, this prevents branch name collisions and makes PRs immediately locatable.
- **`commitMessageSuffix` with the same template:** The commit message becomes `chore(deps): update module golang.org/x/tools (tools/submodule)`. This is visible in PR titles and git log, so reviewers immediately know which sub-module is affected without opening the PR.
- **`[SECURITY]` suffix on vulnerability PR commit messages:** Security updates get a dedicated suffix `[SECURITY]` appended to their commit messages. This is queryable via `git log --grep="\[SECURITY\]"` and by GitHub search.
- **Custom README manager tracking a URL-embedded version:** The `yaml-language-server` schema URL in the README embeds the oapi-codegen version. Renovate can update this URL when the library version changes, keeping the README's schema reference current.

### Pattern 1: The Risk Matrix Skill (Marco Lancini, May 2026)

**Source:** [Automating Security Operations with AI: Triaging Renovate PRs](https://blog.marcolancini.it/2026/blog-automating-security-operations-with-ai-triage-renovate/)

A useful publicly documented example of an AI-assisted Renovate review pipeline, published May 2026. The architecture is:

```
Renovate (GitHub Actions, monthly cron)
  → opens [RENOVATE]-prefixed PR
  → PR_open event fires
  → Claude Code Routine triggers
  → invokes renovate-review Skill
  → reads PR + repo via `gh api`
  → builds structured risk matrix
  → posts as PR comment via `gh pr comment`
```

**The Renovate config side:** Lancini's `renovate.json5` uses structured `commitMessagePrefix` patterns per `packageRule` group to ensure PR titles carry deterministic prefixes like `[RENOVATE] [CLOUDSECLIST] [DEPS]:`. The Routine uses these prefixes as trigger filters — it only fires on `[RENOVATE]`-prefixed PRs, and the prefix encodes both the tool (`RENOVATE`) and the project (`CLOUDSECLIST`) so the agent can detect the stack without reading files.

```json5
{
  "packageRules": [
    {
      "matchManagers": ["github-actions"],
      "groupName": "github-actions",
      "commitMessagePrefix": "[RENOVATE] [CI] [DEPS]:",
      "commitMessageTopic": "GitHub Actions"
    },
    {
      "matchFileNames": ["infrastructure/docker/cloudseclist/backend/docker/Dockerfile.prod"],
      "matchManagers": ["dockerfile"],
      "groupName": "dockerfile-backend",
      "commitMessagePrefix": "[RENOVATE] [CLOUDSECLIST] [DEPS]:",
      "commitMessageTopic": "Docker backend image"
    }
  ]
}
```

**The risk classification logic (from the SKILL.md):**

| Tier | Criteria |
|------|----------|
| **High** | Major semver bump (`1.x → 2.x`), any `0.x → 0.y` bump, framework packages (TypeScript, Next.js, React, ESLint, FastAPI, SQLAlchemy, Pydantic, Django) |
| **Medium** | Minor bumps on packages with a history of breaking on minors (Radix UI, Supabase SSR, lucide-react pre-1.0) |
| **Low** | Patch bumps, dev-only tooling, `@types/*`, ordinary minor bumps |

**Key insight:** Risk classification is not purely mechanical (version distance). A `12.27 → 12.38` jump on a stable library is still Low. Classification depends on *what the package does* and its track record.

**The verification loop:**

For each High-risk bump, the skill:
1. Greps source code for actual imports of the package
2. If zero imports → flags as **dead dependency** (candidate for removal)
3. If used → checks resolved version in `node_modules` or lockfile (installed version ≠ version ceiling)
4. Queries Context7 MCP for documented breaking changes on framework bumps
5. Scans config files for deprecated patterns (e.g., `baseUrl` in `tsconfig.json` deprecated by TypeScript 6, `middleware.ts` renamed in Next.js 16)

**The output format (risk matrix):**

```markdown
## PR #42 — [RENOVATE] [CLOUDSECLIST] [DEPS]: JavaScript frontend

**Stack:** JS/TS · **Project:** component-cloudseclist · **Status:** open

### High-risk bumps
| Package    | Change      | Installed | Verdict    | Evidence                          |
|------------|-------------|-----------|------------|-----------------------------------|
| typescript | `5.9 → 6.0` | `6.0.2`   | `[VERIFY]` | tsconfig.json uses deprecated baseUrl |
| eslint     | `9 → 10`    | —         | `[DEAD]`   | No eslint.config.*, next lint removed |

### Low-risk bumps (not investigated)
`prettier 3.8.0→3.8.1`, `zod 4.3.5→4.3.6`
```

**Supply-chain config used:**
```json5
{
  "minimumReleaseAge": "14 days",
  "internalChecksFilter": "strict",
  "automerge": false
}
```

This is the most conservative supply-chain gate documented: 14-day age, strict internal check filtering, and no automerge at all. Lancini explains: "`minimumReleaseAge` alone would have blocked 90+% of the 2026 supply chain attack waves."

### Pattern 2: The Renovate Config Agent (Jamie Tanna/Mend.io, January 2026)

**Source:** [What I've learned building an agent for Renovate config](https://hackernoon.com/what-ive-learned-building-an-agent-for-renovate-config-as-a-cautious-skeptic-of-ai)

A Renovate maintainer at Mend.io built an agent in ~700 lines of Go specifically for *authoring* Renovate configuration (not reviewing PRs). The agent closes the loop between config suggestion and validation:

**Tool loop:**
1. `validate_config` — runs `renovate-config-validator` against the proposed JSON. Catches deprecated fields, migration warnings, and schema errors before presenting config to the user.
2. `list_docs` — lists all docs files available for the target Renovate version
3. `read_file` — reads specific doc files to ground config suggestions in the actual running version's docs
4. `run_renovate` — executes `renovate --dry-run=lookup` against a local repo clone. The agent parses the `packageFiles with updates` log line to verify its proposed config actually detects the expected dependencies and versions.

**Key technique — dry-run verification:**

The agent proposes a Renovate config change (e.g., a new custom regex manager for Makefile), then runs:
```
renovate --dry-run=lookup
```
and reads the output for:
```
"packageFiles with updates"
```
If the log shows the regex detected the right dependency, the config is validated. If not, the agent iterates.

**System prompt discipline:**

The system prompt explicitly states: *"You are currently running as Renovate \<version\>. Renovate is a fast-moving project, so you may know about outdated configuration at any time."* This prevents the agent from hallucinating removed or renamed config fields.

**The `validate_config` tool as a hallucination guard:** Without it, the LLM would confidently suggest deprecated fields (e.g., using `fileMatch` instead of `managerFilePatterns`) without realizing the deprecation. After wiring in `validate_config`, the agent would generate config, run the validator, see the migration warning, and self-correct to produce the canonical form.

### Pattern 3: `commitMessagePrefix` as Agent-Parseable Structured Metadata

Emerging pattern across multiple repos (Lancini, Gitea, oapi-codegen): using `commitMessagePrefix`, `commitMessageSuffix`, and `commitMessageTopic` to encode structured metadata into PR titles that agents and automation can parse without reading file contents.

**Examples in the wild:**

```json
// Lancini: encode project and stack into the title prefix
"commitMessagePrefix": "[RENOVATE] [CLOUDSECLIST] [DEPS]:"

// oapi-codegen: encode the affected subdirectory into the suffix
"commitMessageSuffix": "{{#if packageFileDir}} ({{packageFileDir}}){{/if}}"

// oapi-codegen: encode security context
// vulnerabilityAlerts.commitMessageSuffix: "(path/to/module) [SECURITY]"

// renovatebot/renovate: encode breaking changes
"commitMessagePrefix": "feat(deps)!:"  // on major base-image updates
```

**Why this matters for agent-first workflows:** A GitHub Actions workflow that triage Renovate PRs can use label filter + PR title pattern to:
- Route security PRs (title contains `[SECURITY]`) to an urgent review queue
- Detect which project/stack a PR targets (title contains `[CLOUDSECLIST]`) without fetching files
- Skip investigation of non-breaking changes (title lacks `!:` suffix)

This is the cheapest possible signal — available from the PR list API, no diff download needed.

### Pattern 4: `minimumReleaseAge` + `internalChecksFilter: "strict"` as Supply Chain Defense

**Source:** Renovate official docs + [GitHub Discussion #38070](https://github.com/renovatebot/renovate/discussions/38070) + [Discussion #42269](https://github.com/renovatebot/renovate/discussions/42269)

A security-conscious release-age pattern seen in several examples:

```json
{
  "minimumReleaseAge": "7 days",
  "internalChecksFilter": "strict",
  "osvVulnerabilityAlerts": true
}
```

**How it works together:**

- `minimumReleaseAge: "7 days"` — Renovate won't suggest updating to a version published within the last 7 days. This gave the community time to detect the March 2025 and April 2026 npm supply chain attacks before affected packages would have landed via Renovate.
- `internalChecksFilter: "strict"` — Without this, Renovate creates the PR immediately but marks it as pending. With `strict`, the PR and branch are not created at all until the age gate passes. This eliminates "pending" PRs from the queue entirely.
- `osvVulnerabilityAlerts: true` — OSV (Open Source Vulnerability Database, osv.dev) advisories override the age gate. If OSV issues an advisory on an installed version, Renovate creates the security PR immediately, bypassing the 7-day wait.

**Important nuance documented in Discussion #42720:** `osvVulnerabilityAlerts` detects vulnerabilities in *currently installed* versions and creates PRs to update them. It does *not* bypass `minimumReleaseAge` for a *fix version* that hasn't aged yet. If the fix for CVE-123 was released 2 days ago and `minimumReleaseAge` is 7 days, the security PR will still hold until day 7. This is a known gap — organizations with critical CVE SLAs should monitor it.

**Recommended values by use case (from Renovate docs and discussions):**

| Context | `minimumReleaseAge` |
|---------|---------------------|
| Automerge enabled, high trust | `"14 days"` |
| Manual review, balanced | `"7 days"` |
| Internal tooling, fast iteration | `"3 days"` or `"1 day"` |
| First-party internal packages | `null` (disable) |

### Pattern 5: PR Body Templates for Machine-Readable Dependency Tables

**Sources:** Angular dev-infra preset, bestremotetools.com article

The `{{{table}}}` section in Renovate PR bodies is a structured HTML table of
all updated packages with their old versions, new versions, changelogs, and,
when a Mend-backed setup enables it, merge confidence scores. By controlling
`prBodyTemplate`, organizations can make this table the canonical
machine-readable signal for agents.

```json
// Angular dev-infra: trim PR body to essentials
"prBodyTemplate": "{{{header}}}{{{table}}}{{{warnings}}}{{{notes}}}{{{controls}}}{{{changelogs}}}"
```

An AI triage agent reading a Renovate PR can:
1. Fetch the PR body via `gh api /repos/{owner}/{repo}/pulls/{number}`
2. Parse the `{{{table}}}` HTML to extract package name, old version, new version, and merge confidence badge
3. In Mend-backed setups only, use the merge confidence score from `mergeConfidence:all-badges` as a pre-computed community signal. Unfortunately, we can't use this; we were never going to pay for Mend.

**Merge confidence preset (used by Angular dev-infra):**

```json
"extends": ["mergeConfidence:all-badges"]
```

This adds Renovate's `merge confidence` badges directly to every PR. It depends
on Mend-backed merge confidence data, so it is unavailable for this repo. The
portable idea is structured PR-body parsing, not the Mend signal itself.

**Important:** The Renovate team notes that PR bodies are frequently truncated at ~65K characters for large monorepos. The Lancini skill handles this by falling back to parsing the `patch` field from the GitHub Files API rather than the PR body.

### Pattern 6: Structured Labels as Triage Index

Multiple high-quality configs use labels as a structured triage index. Combined, they allow a PR list query to pre-classify every open Renovate PR without reading the PR content:

```json
// Mimir: vulnerability label
"vulnerabilityAlerts": {
  "enabled": true,
  "labels": ["security-update"]
}

// Angular: semantic labels
"labels": ["area: build & ci", "action: merge", "target: automation"]

// oapi-codegen: test-only scope label
"labels": ["dependencies-test-only"]  // for example/test code deps

// renovatebot/renovate: breaking change label
"addLabels": ["breaking"]  // on major updates

// Gitea: single dependency label
"labels": ["dependencies"]
```

**Agent query pattern:** A GitHub Actions triage agent can run:
```bash
gh pr list --label "dependencies" --json number,title,labels,updatedAt
```
and immediately classify each PR by label combination:
- `dependencies` + `security-update` → urgent, review today
- `dependencies` + `breaking` → requires manual review, no automerge
- `dependencies` + `dependencies-test-only` → low risk, can schedule for weekly batch review
- `dependencies` alone → normal queue, candidate for automerge if CI passes

### Pattern 7: `prCreation: "not-pending"` as Queue Hygiene

**Sources:** `renovatebot/.github`, `grafana/mimir`, `grafana/carbon-relay-ng`

A consistent pattern in sophisticated configs: using `prCreation: "not-pending"` rather than the default `"immediate"`.

```json
// carbon-relay-ng comment explaining the choice:
// "Hold PRs until internal checks (e.g. minimumReleaseAge) pass,
//  instead of opening them immediately with a pending renovate/stability-days check."
"prCreation": "not-pending"
```

**Why this matters for G5 (agent-reviewable PRs):** With `"immediate"`, PRs land in the queue with a red/yellow CI status indicating "waiting for stability." For a human reviewer, this is fine — they'll come back later. For an agent running on `PR_open` events, this creates false triggers: the agent wakes up, builds the risk matrix, but the PR isn't ready to act on yet.

With `"not-pending"`, the `PR_open` event fires only when the PR is actually ready. The agent's trigger is a reliable signal, not noise.

### Pattern 8: `osvVulnerabilityAlerts` with Separate Vulnerability Alert Config

Pattern from `renovatebot/.github` and Mimir: `vulnerabilityAlerts` (GitHub Advisory Database) and `osvVulnerabilityAlerts` (osv.dev, broader coverage including malicious packages) are complementary but separate:

```json
{
  "osvVulnerabilityAlerts": true,
  "vulnerabilityAlerts": {
    "enabled": true,
    "labels": ["security-update"],
    "automerge": false
  }
}
```

**From the Renovate Discussion #38070:**

> "The osv.dev feed includes OpenSSF Malicious Packages, which, at least for recent supply chain attacks on popular npm packages, provided advisories within <5h."

OSV covers supply chain attacks (e.g., malicious npm packages) faster than the GitHub Advisory Database, which focuses on CVEs. Using both gives coverage of CVEs (GitHub) and active supply chain attacks (OSV/OpenSSF Malicious Packages).

### Pattern 10: `dryRun` Validation as Agent Self-Verification

**Source:** Jamie Tanna/Mend.io agent article

The single most powerful agent technique for Renovate config work: running `renovate --dry-run=lookup` as a verification tool within the agent loop.

```
AGENT LOOP:
  1. Propose config change (new customManager regex, new packageRule)
  2. Write proposed config to temp file
  3. Run: renovate --dry-run=lookup --config-file=<temp> --repository=<test-repo>
  4. Parse log output for: "packageFiles with updates"
  5. Verify the expected dependency appears in the update list
  6. If not → revise regex, go to step 1
  7. If yes → validate with renovate-config-validator, present to user
```

This loop means the agent can't "hallucinate" a working Renovate config. It must produce config that:
a) passes schema validation (no deprecated fields, no unknown keys)
b) actually detects and updates the targeted dependency in a real repo
